# Data retention

Agent Flight Recorder (AFR) retains audit data locally by default. This document describes the retention policy for the self-hosted MVP.

## Storage

- AFR stores runs, events, checkpoints, and state snapshots in a local SQLite file.
- The default database path is configured by the operator at deployment time.
- No data is sent to external services unless the operator explicitly configures an export or remote backend.

## Retention period

- Records are retained indefinitely until the operator deletes the database file or a future deletion endpoint is used.
- There is no automatic expiry, TTL, or garbage collection in the self-hosted MVP.
- Operators are responsible for defining and enforcing their own retention schedule outside of AFR.

## Exports

- Exported run packages (e.g. JSON archives) are the operator's responsibility.
- AFR does not track, expire, or delete exported files automatically.

## Audit event actors

Event payloads may include an `actor` field to identify the source of the event. Supported values are:

- `system` – events generated by AFR itself (e.g. lifecycle transitions).
- `operator` – events explicitly logged by a human operator or administrative action.
- `wrapper` – events recorded automatically by AFR decorators (`record_model_call`, `record_tool_call`).
- `replay` – events emitted during replay or reconstruction.

`actor` is optional. When omitted, no schema change is required and existing events remain valid.
